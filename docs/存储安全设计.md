# 存储安全设计规范

> 最后更新：2026-07-18
> 版本：v1.0

## 一、设计原则

### 1.1 数据与程序分离

程序代码和数据目录**物理隔离**，卸载/重装程序不影响数据。

```
~/Projects/moyin-creator-webui/          ← 程序代码（可随时删除重装）
~/Documents/moyin-creator/data/          ← 用户数据（永不随程序卸载而删除）
```

程序只读写 `data/` 目录，不写入 `~/Projects/`、`~/Library/`、`~/.moyin/` 等位置。

### 1.2 三层防护

| 层 | 机制 | 防止的问题 |
|----|------|-----------|
| **路由层** | pid=null 拒绝写入 + 从数据中提取 projectId 而非信任全局状态 | 切换项目时的竞态写入、setActiveProject(null) 覆盖 |
| **大小门槛** | `safetyCheckBeforeWrite` — 新数据 <100B 且旧数据 >200B 时拒绝 | 空壳 JSON 覆盖真实数据 |
| **恢复层** | `recoverFromLegacy` — 每次启动比较新旧，legacy 富 → 恢复 | 迁移后数据丢失、手动误删 `_p/` 文件 |

### 1.3 不可替换原则

- **重装程序**：用户只需要重新 `git clone` + `npm install`
- **数据不受影响**：`~/Documents/moyin-creator/data/` 不会被任何程序操作删除
- **npm 脚本**：不会包含任何 `rm -rf data/` 或 `clean` 命令
- **存储服务**：启动时只 `mkdir -p`（创建），绝不 `rm`（删除）

## 二、数据目录结构

```
~/Documents/moyin-creator/
├── data/                          ← 程序写入的唯一数据目录
│   ├── _p/                        ← 按项目隔离的数据
│   │   └── {projectId}/           ← 每个项目的独立目录
│   │       ├── script.json        ← 剧本数据
│   │       ├── director.json      ← 导演数据
│   │       ├── media.json         ← 媒体文件索引
│   │       ├── characters.json    ← 角色
│   │       ├── scenes.json        ← 场景
│   │       ├── timeline.json      ← 时间线
│   │       └── sclass.json        ← 分镜
│   ├── _shared/                   ← 跨项目共享数据
│   │   ├── characters.json
│   │   ├── scenes.json
│   │   └── media.json
│   ├── images/                    ← 生成的图片
│   │   ├── scenes/
│   │   └── shots/
│   ├── _migrated.json             ← 迁移完成标记
│   └── moyin-*.json               ← [legacy] 迁移来源，仅作恢复源，不再写入
├── data_backup_YYYYMMDD_HHMM/     ← 备份（手动或自动触发）
└── media/                         ← 用户上传的源文件
```

### 2.1 关键约束

| 规则 | 理由 |
|------|------|
| `_p/_migrated` 只能是 **JSON 文件**，不能是目录 | 存储服务的 `exists` API 会将 `_p/_migrated` 解析为 `_p/_migrated.json` 文件，若同名目录存在则 `fs.existsSync` 返回 `true`，`writeFileSync` 失败 → 迁移flag永远写不进去 |
| legacy `moyin-*-store.json` 只读不写 | 一旦被覆写（哪怕只是空壳），恢复源丢失 |
| `_p/{projectId}/` 必须至少有 `script.json` + `director.json` | 空目录说明数据被清空（E2 bug），应触发 recovery |
| 不做 `rm -rf _p/` 类操作 | 存储服务只增删 JSON 文件，不递归删目录 |

## 三、存储服务 API

### 3.1 端点

| 方法 | 路径 | 作用 |
|------|------|------|
| `GET` | `/api/storage/:key` | 读取 JSON 文件内容 |
| `GET` | `/api/storage/:key/exists` | 检查是否存在 |
| `GET` | `/api/storage/:key/raw` | 读取原始图片数据（用于 `<img src>`） |
| `POST` | `/api/storage/:key` | 写入 JSON 文件 |
| `DELETE` | `/api/storage/:key` | 删除文件 |
| `GET` | `/api/storage/keys/:prefix` | 列出匹配前缀的所有 key |
| `DELETE` | `/api/storage/dir/:prefix` | 递归删除前缀下所有 JSON |
| `GET` | `/healthz` | 健康检查 |

### 3.2 安全措施

- **路径遍历防护**：`safeKey()` 过滤 `..`、`\`、`\0`
- **目录自动创建**：`ensureDir()` → `fs.mkdirSync(dir, { recursive: true })`
- **无删除数据目录**：没有任何 API 支持删除整个 `data/`

### 3.3 启动参数

```bash
node local-storage-server.mjs \
  --port 3001 \                          # 默认 3001
  --data-dir ~/Documents/moyin-creator/data  # 默认路径
```

## 四、前端数据流

### 4.1 启动顺序

```
App 启动
  → initApp() 调用 installWebShims()
    → 注入 window.fileStorage（HTTP API 客户端）
    → 注入 window.imageStorage / imageHostUploader / 等
  → migrateToProjectStorage()
    → 将 legacy moyin-*.json → _p/{projectId}/*.json
    → 成功后写 _migrated.json
  → recoverFromLegacy()
    → 遍历 _p/ 下所有项目，检查是否为空
    → 空则从 legacy 恢复
  → Zustand persist stores rehydrate
    → 各 store 通过 createProjectScopedStorage / createSplitStorage
    → 从 _p/{activeProjectId}/ 读取数据
```

### 4.2 写入路径

```
Zustand persist 自动触发
  → createProjectScopedStorage.setItem()
    → 1. 从数据 JSON 中提取 activeProjectId（避免竞态）
    → 2. pid=null → 拒绝写入（记日志，返回）
    → 3. pid 有效 → safetyCheckBeforeWrite()
      → 新数据 <100B 且旧数据 >200B → 拒绝
      → 通过 → fileStorage.setItem(_p/{pid}/{storeName}, value)
        → HTTP POST /api/storage/_p/{pid}/{storeName}
        → fs.writeFileSync() 写入磁盘
```

## 五、异常处理

### 5.1 已知异常及修复状态

| 异常 | 状态 | 修复 |
|------|------|------|
| `_p/_migrated/` 目录（非文件） | 🔴 需修复 | 存储服务 exists 路由 bug → 修复后需手动清理 |
| `_p/76bacc93` 空目录 | ⚫ 数据已丢失 | 不可恢复，等待修复后不再发生 |
| `_p/a4bbe260` 空目录 | ⚫ 数据已丢失 | 同上 |
| legacy `moyin-*.json` 全为空壳 | 🔴 需处理 | 弱化为只读标记，不再写入 |
| `STORAGE_API_BASE` 硬编码 IP | 🔴 需修复 | 改为 `window.location.origin` （同源）或通过 Vite 代理 |

### 5.2 恢复流程

1. 应用启动 → `recoverFromLegacy()` 运行
2. 遍历 `_p/` 下所有项目目录
3. 对每个项目，检查 `script.json` / `director.json` 是否为空壳
4. 若为空，查找 legacy `moyin-*-store.json` 中该 pid 的数据
5. 从 legacy 恢复到 `_p/{pid}/` 下

## 六、重新安装检查清单

当用户说「卸载重装」时，Agent 应执行：

```
□ 1. 确认 ~/Documents/moyin-creator/data/ 存在且有数据
□ 2. 确认 data/ 大小 > 10KB（非空）
□ 3. 备份：cp -r data/ data_backup_$(date +%Y%m%d_%H%M)/
□ 4. 删除程序：rm -rf ~/Projects/moyin-creator-webui/
□ 5. 重新安装：git clone + npm install
□ 6. 启动：npm run storage（验证 3001 端口）→ npm run dev:web（验证 5174 端口）
□ 7. 验证数据：检查 data/_p/ 下项目目录文件完整
□ 8. 浏览器打开 → 确认项目列表正常、可切换项目
```

## 七、禁止操作

| 禁止 | 原因 |
|------|------|
| `rm -rf data/` 或 `rm -rf _p/` | 数据永久丢失，不可恢复 |
| 写入 `data/` 以外的目录 | 打破数据/程序分离 |
| `setActiveProject(null)` | 触发 pid=null 的 persist write |
| 在存储服务中使用 `fs.rmSync(dir, { recursive: true })` | 可能递归删除数据目录 |
| `fs.writeFileSync` 不加 `ensureDir` | 父目录不存在时静默失败 |
